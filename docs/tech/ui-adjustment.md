# UI 界面調整可行性

CLAUDE.md「其他需求」要求獨立討論 UI 界面調整的可行性。本文聚焦「版面/座標/縮放」層級的調整選項,
與 `docs/tech/cjk-screen-rendering.md`(渲染管線)、`docs/tech/screen-spec-info-research.md`
(單一畫面的座標/TSV 盤點)互補——那兩份談「怎麼把中文畫上去」,本文談「要不要、能不能動原版版面」。

## 1. 現況

- **原版 640×480 固定邏輯畫布**:`openorion2` `screen.h` 與本專案 `cmd/moo2/main.go:27` 都以
  640×480 為邏輯座標系,`game.Layout()`(`main.go:60`)、`helpGame.Layout()`(`help.go:47`)一律
  回傳固定的 `helpScreenW/helpScreenH`(640×480),不隨外層視窗尺寸變動。目前也沒有呼叫
  `ebiten.SetWindowResizable`,視窗大小在 `SetWindowSize` 呼叫時定死(`main.go:275`、
  `help.go:187`),等同重現原版「不可縮放單一解析度」的行為。
- **兩種文字呈現手法並存,對應兩種版面自由度**:
  - **烘焙點陣圖 + 疊字覆蓋**(`cmd/moo2/overlay.go`):按鈕/標籤英文字畫死在 `.lbx` 點陣圖裡,
    中文化用「採樣底色擦除 + 置中疊繪」(`overlayGame.Draw`,`overlay.go:61-85`),座標完全繼承
    原版 `labelRect{x,y,w,h}`(如 `overlay.go:17-22`)。這類畫面**版面不可調**,能動的只有文字內容
    與字級。
  - **自繪面板 + 動態文字層**(`cmd/moo2/help.go` 為首例,`docs/tech/cjk-screen-rendering.md`
    記錄的樣板):面板底色/框線自己畫,文字走 `i18n.Registry` + `uifont.WrapText` 動態換行,
    座標是我們自己定的常數,不受原版點陣圖束縛。這類畫面**版面可調整**,只是目前六個示範畫面
    (menu/help/inforeview/raceinfo/colonyview/diploview)仍**沿用原版數值座標**,尚未真正利用
    這個自由度重排版。
- **多數尚未完整實作的畫面**(`screen-spec-info-research.md` 盤點的 `HistoryGraphWidget`/
  `TurnSummaryWidget`/`DocsWidget` 等)目前只有標題+空框,是版面決策成本最低的落腳點——
  這些畫面補實作的同時就能一併決定新版面,不必先重排一個已完工的舊版面。

## 2. UI 調整選項(逐一列可行性與取捨)

### 選項 A:保留原版版面 + 疊字(現行主線做法)

座標、按鈕排列、面板大小全部照抄原版(`overlay.go` 已是此路線)。

- **優點**:保真度最高,老玩家開圖秒認得;工作量最小,座標直接抄 openorion2 原始碼常數,不必
  重新設計互動邏輯;和已完成的截圖 before/after(`docs/reference-screens.md`)一致,方便對照驗收。
- **缺點**:中文平均字寬 > 英文(尤其全形標點、長科技名),在原版按鈕/標籤的固定框內容易溢出或
  被迫縮到難以辨識。`screen-spec-info-research.md` §6 已指出 `TechListWidget` 的 `fitText`
  縮字距擠壓策略「若直接沿用可能造成中文標題被過度壓縮」,是本選項最直接的副作用。
- **取捨**:適合按鈕類短標籤(2-4 字,如「確認」「取消」)——這類詞中文通常比英文原文更短,
  溢出風險低;不適合長句/長清單項(科技名、種族特性描述),需搭配「縮字」或「局部放寬邊界」處理。

### 選項 B:縮字 vs 拉畫布(對照 retro CJK 經驗)

當中文在原版框內放不下時,有兩條路,兩者互斥、需逐元素判定:

- **縮字**:降低該元素字級直到塞進原框。優點是版面座標完全不動,和選項 A 相容;缺點是字級下限
  受 CJK 點陣/描邊字型可讀性限制(過小易糊,尤其疊在原版彩色底圖上要保留描邊/陰影才能辨識,
  參考前作 `master-of-orion` 踩過的「全域字級寫死 14px、大按鈕字太小」教訓,
  見 `docs/kickoff/03-button-cht.md` §1)。
- **拉畫布(局部放寬邊界)**:原框太窄就把框本身放大(如 `drawInfoBox` 寬度、`TechListWidget`
  面板寬度),但這只在**自繪面板**類元素上可行(選項 A 的烘焙點陣圖按鈕框放大了背景圖並不會跟著變,
  會露出圖外空白或壓到相鄰元素)。因此「拉畫布」實質只對本專案自繪的部分開放,是選項 A 的例外
  補丁,不是全域策略。
- **建議判準**:先量測——`uifont.Font.Measure` 對每個候選中文字串在目標字級下的實際寬度,超出原框
  多少決定用縮字或拉框;`docs/tech/cjk-screen-rendering.md` 已有 `WrapText` 可重用的量測介面,
  可以在自繪面板批次跑一次「原框寬度 vs 中文譯文寬度」比對表,取代逐畫面臨場目測。

### 選項 C:現代化調整(可縮放視窗、高 DPI、重排版面)

放開 640×480 固定假設,讓視窗可拖曳縮放、支援高 DPI 螢幕、必要時重排版面(如加寬側欄、改用
響應式清單而非固定列數)。

- **技術可行性**:ebiten 的 `Layout(outsideWidth, outsideHeight)` 本來就是為此設計——回傳值是
  **邏輯畫布尺寸**,與外層視窗實際像素尺寸(可能因 DPI/使用者拖曳而不同)分離,ebiten 引擎層自動
  處理縮放/取樣,不需要我們手刻縮放邏輯。目前 `Layout()` 固定回傳 640×480,是**選擇**而非限制;
  改成可縮放本身工作量不大(`ebiten.SetWindowResizable(true)` + `Layout` 回傳邏輯尺寸不變,只是
  縮放係數交給 ebiten 處理整數/非整數縮放)。真正的成本在**畫面內容要不要跟著重排**。
- **兩個子選項**:
  1. **只縮放,不重排**(邏輯畫布仍 640×480,视窗變大只是等比放大顯示):成本最低,`Layout()` 不變
     即可,ebiten 會用線性/最近鄰縮放整個畫布。適合先做,驗證使用者是否真的需要更大視窗。
  2. **真重排**(邏輯畫布尺寸隨視窗變、UI 元素用相對版面而非寫死座標):成本高——選項 A/B 討論的
     所有座標常數(`labelRect`、`drawInfoBox` 等)都要改成相對佈局或至少參數化,等同重寫版面系統,
     且會讓「保真度」大幅下降(原版本來就是為 640×480 CRT/VGA 比例設計,重排後不再是「原版加中文」
     而是「新介面」)。
- **取捨**:保真 vs 易用性的核心矛盾在這裡最明顯。子選項 1(只縮放)幾乎零風險、建議直接做;
  子選項 2(真重排)建議列為**遠期/可選**項目,且應該只套用在「本來就沒有原版版面可循」的新增畫面
  (如尚未實作的 `TurnSummaryWidget`),不動已有精確座標對照的既有畫面。

### 選項 D:烘焙文字按鈕的中文化(去文字化素材 + 疊中文)

`docs/kickoff/03-button-cht.md` 已整理前作(`master-of-orion`/`master-of-magic`)驗證過的兩種手法,
本專案 `overlay.go` 目前用的是其中「擦底疊字」一種:

1. **擦底疊字**(現行):採樣底色蓋掉英文區,不改變按鈕外框大小,疊置中文。
   - 優點:不需要美術資源,`overlay.go` 已跑通、有截圖驗證(`docs/reference-screens.md` 主選單/
     行星列表案例)。
   - 風險:採樣點若剛好落在漸層/花紋底圖上,擦除色塊會與周圍不連續(`reference-screens.md` 已記錄
     一個已知瑕疵:「顯示篩選」下方粗體英文單一採樣色擦不乾淨,邊緣微透)。這是選項 D 最大的技術債,
     優先度高於視覺美化類調整。
2. **整張圖替換**(`image_override.go` 模式,mom 已驗證,本專案尚未用到):在圖片載入入口攔截,
   外部目錄提供 `<lbx>_<index>.png` 中文版即替換,原始版權 `.lbx` 不動。適合「圖文一體、擦除代價高」
   的按鈕(如帶裝飾花紋的標題橫幅)。
3. **IMGLOG 探查**(前作驗證有效,本專案 §7 已提及「待截圖確認是否烘焙文字」的多組按鈕,
   `screen-spec-info-research.md` 提到 `ASSET_TECHSEL_AREA_BUTTONS`、`ASSET_INFO_*_BUTTON`、
   `ASSET_INFO_ACHIEVEMENTS_BUTTON` 皆屬此類):加一個開關記錄每次 `(lbx, index)` 載入請求,
   一次性列出「哪張圖對應畫面上哪個 UI 元素」,取代逐畫面臨場截圖判斷,建議在批次盤點按鈕前就做。

- **取捨**:手法 1(擦底疊字)是預設路線,足以應付版面不變的多數按鈕;手法 2(整張圖替換)只在
  手法 1 擦不乾淨或圖文一體時才啟用,避免無謂的美術產出成本;手法 3(IMGLOG)是效率工具,不是
  最終呈現手法,建議在進入按鈕密集的畫面(研究彈窗、艦艇設計)前先跑一次。

## 3. 關鍵痛點

- **中文寬度**:中文平均字寬明顯寬於等寬拉丁字元,原版為英文設計的按鈕寬度/欄位寬度是最大的結構性
  限制。任何新畫面盤點都應先做「原框寬度 vs 中文譯文預期寬度」量測(見選項 B 建議),而不是實作
  完才發現溢出。
- **烘焙文字**:凡是「圖」而非「字串表」的英文,翻譯表清單改了也沒用(`docs/kickoff/03-button-cht.md`
  §1 的核心事實)。這類元素必須先用 IMGLOG 或逐一截圖盤點,才能決定用選項 D 的哪種手法,不能假設
  `misc.tsv` 一定涵蓋。
- **固定座標**:選項 A(保留原版版面)下,所有座標都是 openorion2 原始碼裡的字面常數,牽一發動全身
  ——放寬一個元素的框可能压到緊鄰元素(`screen-spec-info-research.md` §6 已提醒清單類元件的
  「列距/選擇框高/可視區總高」三者要一起換算,前作艦船設計清單踩過這個坑)。任何局部調整前先確認
  相鄰元素的邊界,不要只看單一元素本身。
- **多解析度**:目前完全沒有多解析度支援(`Layout()` 寫死 640×480、視窗不可縮放)。這不是 bug,
  是刻意跟隨原版行為的現狀,但也代表「使用者要求可縮放視窗」這類需求目前完全沒有退路——選項 C
  子選項 1(只縮放不重排)是最小可行的回應方案。

## 4. 建議

- **分階段,保真優先、必要處局部調整**,不整體重排:
  1. 先修複選項 D 已知的擦除瑕疵(行星列表「顯示篩選」色塊不乾淨),技術債優先於新功能。
  2. 新畫面(`TurnSummaryWidget` 等尚未實作、無精確原版座標對照的空框)用自繪面板從一開始就用
     `uifont.WrapText` 動態版面,不追求對齊原版點陣圖座標——這類畫面沒有「原版版面」包袱,
     用選項 B 的量測流程直接設計中文友善的版面。
  3. 已有精確原版座標對照的畫面(按鈕、標籤類),維持選項 A(保留原版版面)為預設,搭配選項 B
     的縮字/局部拉框逐元素處理溢出,不做整體重排。
  4. 選項 C 的「只縮放不重排」風險低、獨立於其他工作,可在任何時間點加(`SetWindowResizable`
     + 保持 `Layout()` 回傳邏輯尺寸不變),建議列入 WORKLIST 作為使用性改善的低成本項目;
     「真重排」(子選項 2)不建議在保真度優先的現階段執行,列為遠期可選項。
  5. 按鈕密集的新畫面(研究彈窗、艦艇設計、外交)開工前,先跑 IMGLOG 或截圖盤點,一次判定
     每個按鈕圖是否烘焙文字,避免逐一臨場判斷造成同一批畫面手法不一致。
