### 1.2 全檔字級普查

`grep '.Draw(\|.DrawCentered('` 全檔統計,實際出現的 size 值:
**10, 11, 12, 13, 14, 15, 16, 17(未見)、20, 22**(銀河議會標題)。共 8 種離散值,
分布在約 30 處呼叫點。

**關鍵發現(修正原始問題的框架)**:並非真的「全畫面死一個字級」,而是**同一畫面內,不同語意
層級的元素常共用同一 size**——最明顯的例子是研究畫面:畫面標題「選擇新研究」與 8 個並列的領域
分類標題(建設/動力/化學…)在程式碼裡是**同一個 `defSize=13`**(`interactive.go:2066-2068`,
兩者的 `labelRect.size` 都是 `0`,回退到同一個 `defSize`),沒有做「標題 > 分類頭」的字級區分,
即使畫面版面(標題橫跨全寬 vs 分類頭在窄欄內)造成視覺上有落差,字體本身沒有真正放大。
殖民地總覽的表格欄位標題與資料列同樣是 13/13。這是造成「均一感」的根源,而不是專案裡真的只用了
一種 px 值。

### 1.3 按鈕/標籤覆蓋手法:兩套並存機制

程式碼裡實際存在**兩套**「蓋掉烘進圖英文」的機制,兩者忠實度不同:

1. **`overlayScreen` + `samplePlate`**(`interactive.go:114-195`,主選單/星系工具列/殖民地
   表格/研究畫面等多數畫面用此):擦除底色**不是**寫死的深色,而是在標籤範圍**上下緣 + 左內緣**
   取樣像素、算**亮度中位數**當底色(`samplePlate` 函式,`interactive.go:168-195`),再用
   `vector.DrawFilledRect` 填單色矩形、置中疊字。這個機制會依附近底圖顏色調整,比純黑底貼合度高,
   但仍是**純色塊**(不含底圖的漸層/紋理/浮雕明暗),且完全沒有描邊或陰影。
2. **`tacticalScreen.drawBarLabelsCHT`**(`interactive.go:1443-1454`,戰術戰鬥控制列專用):
   **繞過**上述 1 的採樣機制,對全部 7 個按鈕一律填**同一個固定深灰色** `{40,44,54,255}`
   + 淺灰描邊,不論按鈕原本底色為何。程式註解原文即為
   「在原版控制列的英文按鈕上疊深色底 + 中文字,蓋掉烘進的英文」——**這就是使用者截圖看到「貼上去
   感」最重的一處**,因為它連機制 1 的「採樣貼合」都沒有,是純粹寫死的色塊 + 描邊框,與其餘
   COMBAT.LBX 按鈕的金屬質感完全脫節。
3. `info.lbx` 等少數畫面用 `overlayScreen.eraseColor` **強制**指定單色(如 `black := {0,8,24,255}`,
   `interactive.go:1949-1951`),因為背景太均勻、取樣容易誤取到字色本身——這是機制 1 的**顯式
   覆寫**,仍在同一套框架內,不是第三套獨立機制。

### 1.4 字型渲染機制

`internal/uifont/font.go` 確認:
- 字型經 `-font` flag 執行期載入(`.ttf/.otf/.ttc`),`opentype.Parse`/`ParseCollection` 解析,
  依 `docs/kickoff/04-font-choice.md` 定案為 Noto Sans CJK(向量字)。
- `Draw`/`DrawCentered`(`font.go:65-88`)**只有純色填字**,呼叫 `text.Draw` 一次,**沒有任何
  描邊/陰影/發光路徑**——目前整個程式碼庫沒有一處疊字帶立體感效果,方案 (c)(見下)在現有 API
  上完全空白,需要新增。
- 用 `xfont.HintingFull` + ebiten `text/v2` 向量 rasterize,**在目標 px 直接畫**,無 supersample、
  無 nearest-neighbor 選項。`font.go:6-7` 註解明講「取代 mom 手動 rasterize+supersample 的做法」。
  **這與 `docs/kickoff/02-cjk-strategy.md` §3.5「[HARD] supersample,不『小字放大』」的斷言不一致**
  ——實際落地並未 supersample,是直接向量 rasterize。本文不在此修正該文件(超出本次交付範圍),
  但依 CLAUDE.md「每輪盤點文件衝突」的要求在此點名,建議下一輪把 `02-cjk-strategy.md` §3.5
  改成反映實際做法(向量直繪已足夠銳利,supersample 非必要),避免這條過期 HARD 規則繼續佔位。

## 3. 按鈕中文字覆蓋問題:方案比較

### 3.1 四個方案

| 方案 | 說明 | 忠實度 | 工程量 | 風險 |
|---|---|---|---|---|
| (a) 底色取樣按鈕自身像素/漸層 | 現行 `samplePlate`(中位數採樣)已是此思路,但**戰鬥控制列尚未套用**,仍用寫死深色塊 | 中(貼合局部色調,但仍是純色塊,不還原漸層/浮雕) | **小**——把 `drawBarLabelsCHT` 改呼叫既有 `samplePlate`/`overlayScreen` 機制即可,不需新建基礎設施 | 低,`samplePlate` 已有已知瑕疵(單一採樣色在花紋底圖擦不乾淨,`ui-adjustment.md` 已記錄),但屬於既有已知技術債,非新風險 |
| (b) 找不含英文的按鈕底圖 | 若 COMBAT.LBX/MAINMENU.LBX/BUFFER0.LBX 存在按鈕的「乾淨底框幀」(如 hover/disabled 狀態剛好無字),直接借用取代整格 | 若存在則**最高**(保留完整原版材質) | **待查**——需先做 IMGLOG 盤點(`docs/kickoff/03-button-cht.md` §2.5 已建議但本專案尚未執行)才能估工程量 | **高,待查 LBX**:不確定原版按鈕是否真的有無字幀存在。MOO 系列多數按鈕文字是烘進單一幀,不像部分遊戲有獨立的 idle/hover 素材幀分離——**本文不臆測答案**,需實測 `lbxinfo`/IMGLOG 逐格比對後才能判定可行性 |
| (c) 中文字加描邊/陰影貼合浮雕感 | 取代純色擦底,改用描邊(outline)或右下陰影模擬原版按鈕字的立體感 | 中(向量字加描邊可模擬部分立體感,但無法真正還原原版烘焙的多層明暗) | **小**——`uifont.Font` 目前完全沒有描邊/陰影路徑(`font.go:65-88` 只有純色 `Draw`/`DrawCentered`),需新增一個等效函式(多次位移疊繪或用 ebiten 疊圖模擬描邊),屬套件內加法,不影響既有呼叫點 | 低,但**不能單獨取代擦底**——英文仍烘在底圖上,只加陰影不擦除會透字,必須與 (a) 或 `eraseColor` 併用 |
| (d) 主選單 LCD 面板改發光綠字 | 針對「字即發光」的綠色 LCD 面板,現況已是綠字(`{104,224,96}`)+ samplePlate,但**沒有發光/暈光效果**,是純色平塗 | 中高(若做出暈光層,質感更貼近原版 CRT/LCD 發光字) | **中**——ebiten 沒有內建 blur,需手刻(多次半透明位移疊繪模擬模糊)或降採樣近似,比 (a)(c) 複雜 | 中,效果需反覆截圖調校,模糊半徑/透明度沒抓好會糊字,且只適用主選單這類 LCD 風格面板,不是通用方案 |

### 3.2 推薦預設方案

**(a) + (c) 併用,作為所有「烘字按鈕」統一疊字手法的新預設**:

- 先把 `tacticalScreen.drawBarLabelsCHT` 的固定深色矩形改成呼叫既有 `samplePlate` 機制(或直接
  讓 `tacticalScreen` 複用 `overlayScreen` 的擦底邏輯),**消除目前兩套不一致機制並存的狀態**——
  這是成本最低、直接對應使用者「戰鬥控制列」抱怨的一步。
- 在 `uifont.Font` 新增一個帶描邊/陰影的畫字函式,套用在所有疊字按鈕(主選單、工具列、戰鬥控制列),
  補上原版烘焙字具有而現行純色填字沒有的立體感。
- 理由:兩者都不需要新美術資源(方案 b 的前提尚未確認存在),工程量小,且都是在既有機制上擴充而非
  重寫;方案 (b) 列為**後續加分項**——IMGLOG 盤點做完、若確認某些按鈕真的有乾淨底幀,可以針對那些
  按鈕單獨升級到 (b),不必等全部按鈕都有答案才能動工,兩條路線不互斥。
- 方案 (d) 只用於主選單 LCD 面板一處,列為獨立的美術加分項,不建議現階段投入(工程量/風險都高於
  (a)(c),且只影響一個畫面)。

## 4. 點陣 vs 平滑字型的取捨

- **現況**:主字型 Noto Sans CJK(向量),`golang.org/x/image` 向量 rasterize + `HintingFull`,
  無 supersample、無 nearest-neighbor 選項(見 1.4)。`docs/kickoff/04-font-choice.md` 已列出
  Cubic 11 / Fusion Pixel / Zpix 等像素字型候選,並明確排出「先驗證 Go 能否解析 → A/B 截圖比較」
  的待辦——**這個 A/B 待辦目前尚未執行**(repo 內只找到 Noto 的載入路徑,沒有像素字型的驗證紀錄)。
- **這是本次違和感的第二個來源**(第一是覆蓋手法):向量字邊緣抗鋸齒平滑,疊在 1996 年點陣美術
  (LBX 底圖、按鈕浮雕)上天生「太新」,不論字級多合適都會有這層違和。
- **低成本折衷(建議先做)**:不需要重寫渲染管線——`-font` flag 本身是通用機制,可以直接指向
  Cubic11/Zpix 的 TTF 檔案。先執行 `04-font-choice.md` 已列的 A/B 待辦,把同一批遊戲文字
  (標題/按鈕/表格內容)用 Noto 與候選像素字型各截一輪圖比對:
  - 若像素字型在 T1–T3(13px 以上,標題/按鈕)清晰可讀且與原版質感更貼近,**只在這些較大字級**
    換成像素字型,T4–T5(12px 以下,表格內容/小字)保留 Noto 向量字確保可讀性下限——這是分級
    用字,不是全有全無的抉擇,直接延伸 `04-font-choice.md` 既有的「Noto 打底、像素字當 A/B」邏輯。
  - 若像素字型在任何字級都因繁體字筆劃多而糊字,則維持全 Noto,只靠第 3 節的描邊/陰影手法補償
    立體感,不勉強上像素字。
- **更徹底的折衷(僅在上述仍不滿意時做)**:自製點陣 CJK atlas(`rulebook/81-retro-cjk-hires-canvas.md`
  提到的 `build_cjk_font.py` pipeline,TTF → 點陣 atlas 子集),徹底關閉抗鋸齒。這屬於**大**工程量
  選項,且會牽動整個 `uifont` 套件的繪字路徑(目前完全基於 `text/v2` 向量 face),不建議在驗證
  「像素字型 TTF 直接替換」是否已經夠用之前就投入。
