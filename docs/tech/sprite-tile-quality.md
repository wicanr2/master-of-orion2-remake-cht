# Sprite / Tile 畫質優化與重繪可行性

> 對應 `PLAN.md` Phase 8、`WORKLIST.md`「sprite/tile 畫質優化可行性 markdown」。討論範圍:原版點陣素材要不要放大、怎麼放大、放大到什麼程度,以及與 CJK 疊字的畫質互動。**本文只列可行性與取捨,不下最終施工排程**——實際排期在 `WORKLIST.md` 逐項認領。

## 1. 現況

### 1.1 原版素材規格

- 容器:`.lbx` 資產封存檔,像素資料為 **8-bit 調色盤索引**(0–255),搭配 scan-line RLE 壓縮或未壓縮兩種存法,細節見 `docs/tech/lbx-format.md`。
- 調色盤:每色 4 bytes,R/G/B 為 **6-bit VGA 值(0–63)**,載入時 `<<2` 展開成 8-bit(0,4,8,…,252)——這代表色彩本身在源頭就只有 64 階,不是現代 24-bit 全彩,`<<2` 後仍有色階跳動(未做 `<<2 | >>4` 補值,本專案刻意保留原版行為)。
- 畫面邏輯尺寸:**640×480**。這點與本專案先前的其他 retro 中文化案例(如 320×200 的 FreeSynd、1oom/MOO1)不同——MOO2 在 1996 年发行時已是當代少見的高解析度 VGA 遊戲,不是「螢幕太小塞不下中文」的典型案例。
- Sprite/UI 圖多半**無內嵌調色盤**,共用同畫面背景圖的全域調色盤(`lbx-format.md` §2.3.1)。這代表任何素材處理管線動到調色盤,都要照著「背景圖決定調色盤、其餘 sprite 沿用」的既有依附關係走,不能單獨換掉某張 sprite 的色表。
- 動畫以 frame 序列表示(`frameCount` + `frameTime` 毫秒),同一 asset 內部通常只有相同尺寸的多幀,不同 UI 元件(艦船圖示、按鈕、頭像、星球)各自獨立 asset、各自尺寸,沒有統一的「tile size」概念——MOO2 不是磚塊地圖遊戲,素材更接近「拼貼式點陣圖示庫」。

### 1.2 目前 ebiten 渲染方式

`cmd/moo2` 目前所有畫面(`main.go`、`galaxy.go`、`help.go`、`colonyview.go`、`raceinfo.go`、`diploview.go`、`overlay.go`)一致採 **640×480 原尺寸邏輯座標**,`Layout()` 直接回傳 640×480,`ebiten.SetWindowSize` 也設為同尺寸——**目前完全沒有內部放大或縮放邏輯**(無 `GeoM.Scale`、無自訂 `Filter`,解碼出的 `image.RGBA` 直接 `ebiten.NewImageFromImage` 後 1:1 blit)。

這與 `docs/tech/cjk-screen-rendering.md`、`internal/uifont/font.go` 的既有結論一致:MOO2 原生解析度已經是 640×480,中文字用 `text/v2` 向量字型在原尺寸直接 rasterize,**不需要**像 320×200 那類遊戲先拉高內部畫布才有空間畫 24×24 點陣字(`rulebook/81-retro-cjk-hires-canvas.md` 的典型情境在此不成立,見第 3 節說明為何仍相關)。

## 2. 優化路線(逐一評估)

以下四條路線不是互斥的里程碑,而是可疊加的**素材處理層級**——原尺寸是基準,其餘路線是在此之上疊加的可選加工。

### 路線 A:原尺寸像素完美呈現(保真基準)

- **做法**:維持現況,LBX 解碼像素 1:1 繪製,不做任何縮放或濾波。
- **優點**:保真度最高——每個像素的色彩索引、RLE 透明區、調色盤映射都可與原版逐位元組比對(`lbx-format.md` 已有 BEAMS/COLREFIT/GAME 三檔全量驗證數據),是後續一切「有沒有走樣」的基準線(oracle)。工作量近乎零,現況已如此。
- **缺點**:現代顯示器多為 1080p/1440p/4K,640×480 視窗在大螢幕上偏小、周圍留白多;直接用作業系統視窗縮放會用 bilinear 糊化像素邊緣(取決於視窗管理器/合成器行為,ebiten 本身不介入)。
- **適用**:任何時候都應保留這條路線作為**驗證基準**與最保守的預設值,不因為其他路線存在就棄用。

### 路線 B:整數倍放大(2x/3x,nearest-neighbor)

- **做法**:內部畫布或 `ebiten.DrawImageOptions.GeoM.Scale(2, 2)` 搭配 `Filter: ebiten.FilterNearest`(ebiten 預設濾波即為 nearest,不需額外指定),640×480 → 1280×960 或 1920×1440。
- **優點**:整數倍放大不產生非整數取樣的鋸齒不均勻(`rulebook/81` 強調「非整數放大即使 nearest 也會出現不均勻 pixel」),像素邊緣維持銳利、色塊分明,符合 1996 年 VGA pixel-art 的原始美術意圖。工作量小——ebiten 原生支援 `GeoM.Scale`,不需要素材離線處理,執行期一行設定即可全域套用。
- **缺點**:窗口變大後,原本就偏小的 UI 文字標籤(原版英文小字)、邊框裝飾線的鋸齒感會被等比放大,不會變得更清晰,只是更大顆——這是 nearest 放大的本質限制,不是 bug。
- **與 CJK 的關係**:因為 MOO2 UI 本來就有 640×480 的可用空間,中文字不需要靠放大內部畫布才擠得下(不同於 `rulebook/81` 的 320×200 典型案例)。這裡放大是**純顯示层級**的選擇,中文字型可以獨立於底圖放大倍率、用向量字型在放大後的座標系直接以合適大小 rasterize,不必跟著底圖一起做整數倍縮放(向量字型在任意目標像素尺寸都能重新算 hinting,不會像點陣底圖那樣被 nearest 放大而鋸齒化)。
- **工作量**:小。屬於執行期渲染設定 + 視窗座標系調整,不改動素材檔案本身。

### 路線 C:AI 超解析度放大(ESRGAN / xBRZ 類離線預處理)

- **做法**:把 LBX 解碼出的原始點陣圖離線跑過 xBRZ(像素藝術專用邊緣感知放大演算法)或 ESRGAN 類神經網路超解析度模型,產出 2x/4x 放大版本存成新素材檔,執行期直接載入放大後的圖。
- **優點**:相較 nearest 硬放大,xBRZ 一類演算法會做邊緣平滑與色塊插值,傳統評測中對「乾淨」的像素圖(如 SNES 時代美術)效果不錯;ESRGAN 類模型對高對比細節（如艦船輪廓)有機會重建出接近手繪的邊緣。
- **缺點與風險**:
  - **失真/風格偏移**:MOO2 素材是 8-bit 調色盤 + 6-bit VGA 色彩,色彩本身精度有限,加上不少 UI 元素刻意用 dithering(抖色)模擬漸層——這類演算法對 dithering pattern 容易誤判成雜訊而抹平或產生非預期的色塊/暈光(ringing artifact),等於**竄改了原始美術決策**,不是單純提高解析度。
  - **一致性風險**:同一 asset 內的多個 frame(動畫序列)若逐張獨立跑放大模型,相鄰 frame 間可能出現放大結果不一致(閃爍感);需要對整個動畫序列做時間一致性處理,複雜度顯著上升。
  - **調色盤依附關係被打破**:第 1.1 節提到多數 sprite 共用背景圖的全域調色盤,若放大流程在 RGBA 階段處理(而非索引階段),放大後的圖已脫離原始調色盤索引,後續若要支援「調色盤 variant」(原版功能,`lbx-format.md` §2.6 提到的設計)就必須維護兩份平行的素材(索引版 + 放大版),資產管理成本增加。
  - **保真爭議**:相對「加工出跟原版設計者意圖不同的畫面」,依 `rulebook/83`(完整性/保全歷史優先),任何用 AI 生成的替代美術都需要清楚標示「非原版素材、屬本專案再詮釋」,不能與原版逐位元組解碼結果混淆呈現,否則會誤導後續想拿本專案素材做進一步逆向或史料考據的人。
- **工作量**:中至大——需建立離線批次處理管線(對所有 LBX 逐 asset 逐 frame 跑放大)、人工抽樣審核輸出品質(尤其 dithering 密集的漸層區域與半透明邊緣)、素材版本管理(原始 vs. 放大版並存)。
- **適用時機**:若要用,建議限定在**低動態、單張靜態的大圖**(如主選單背景、種族頭像、勝利/失敗畫面),避開動畫序列與小尺寸密集圖示(圖示本身像素數少,超解析度模型更容易幻覺出原本不存在的細節)。

### 路線 D:向量化/人工重繪(最高品質、最高風險)

- **做法**:找美術人員或用向量繪圖工具,依原版點陣圖為參考,重新手繪或向量化(SVG)整組素材,以任意解析度輸出。
- **優點**:理論上品質上限最高、可無損適配任何解析度,且能修正原版本身受限於調色盤/解析度而妥協的細節。
- **缺點**:
  - **工作量巨大**:MOO2 素材涵蓋數百個 LBX 資產、數千個 frame(艦船零件組合、種族頭像、建築圖示、UI 邊框等),全套重繪是與遊戲開發規模同等級的美術工程,遠超中文化/移植專案的合理範圍。
  - **易偏離原版**:重繪必然摻入繪者主觀詮釋,即使「盡量貼近原版」,線條粗細、光影邏輯、色彩選擇都會與原始美術師的實際決策產生落差——這與「保全歷史」的專案宗旨（`rulebook/83`）方向相反：重繪產出的是「一個新美術風格的 MOO2 like 遊戲」，不是「MOO2 的忠實移植」。
  - **版權/授權複雜度**:即使是「參考原圖重繪」，實務上仍涉及對原始美術的高度依附，法律與社群觀感風險都高於單純的格式轉換或整數倍放大。
- **工作量**:巨大,且不是一次性投入——後續每次發現重繪版與原版行為有出入(例如某 frame 在特定調色盤 variant 下該用哪個索引)都要重新對照原版排查。
- **適用時機**:不建議整體採用。若確有需求，僅限**全新原創 UI 元素**(本專案自行新增、原版沒有對應物的介面,例如語言切換選單本身的裝飾)——這類元素沒有「原版」可比對,重繪或新繪不涉及「篡改歷史素材」的疑慮。

## 3. 調色盤與 CJK 疊字的畫質互動

- **色彩精度落差**:原版 6-bit VGA 調色盤(64 階/色)與現代 CJK 字型的抗鋸齒(anti-aliasing,通常算在 8-bit/24-bit 色空間下)放在同一畫面時,字型邊緣的漸層灰階會比底圖任何色彩過渡都細膩——這種「底圖粗顆粒、文字細膩」的反差,在原尺寸(路線 A)最不明顯(因為整體都小,肉眼較難分辨顆粒),放大到 2x/3x(路線 B)後反差會更清楚地浮現,但這是**忠實呈現原版限制**,不是移植引入的瑕疵。
- **可讀性**:目前 `internal/uifont` 用 `text/v2` 向量字型依目標像素尺寸直接 rasterize(非點陣 bitmap font),CJK 字型本身抗鋸齒品質不受底圖調色盤限制,可獨立維持清晰——`docs/tech/cjk-screen-rendering.md` 記錄的百科檢視器已驗證此路徑可行。這代表**中文可讀性問題目前不存在**,不需要靠放大底圖來解決,`rulebook/81` 描述的「縮小字 vs 拉畫布」兩難,在 MOO2 這個已是 640×480 原生高解析度的案例中並不成立。
- **對比與疊字位置**:原版部分 UI 面板背景本身較暗或帶紋理(如深色面板、羊皮紙風格對話框),中文字疊加其上時對比是否足夠,需要在實際畫面(而非孤立截圖)中逐一檢視——尤其若未來採路線 C 做背景放大,dithering 若被平滑掉會改變原本的明暗分布,可能連帶影響文字對比,這是路線 C 需要人工審核的另一個理由。
- **小結**:CJK 疊字品質目前**與底圖放大策略解耦**——不論路線 A/B/C 選哪個,文字渲染路徑都不必跟著改,這是本專案優於「先天解析度不足只能靠拉畫布擠字」類案例的既有優勢,應維持這個解耦設計,不要讓底圖放大邏輯滲透進文字渲染路徑。

## 4. 建議

依 `rulebook/83`(完整性優先,不以投報率為由放棄或簡化),下述分階段建議是**施工順序**而非「先做值得的、後做不值得的」篩選——四條路線最終都值得存在,差別只在於先後與適用範圍。

1. **第一階段(現況即達成)**:路線 A 原尺寸像素完美呈現,作為永久保留的保真基準與驗證 oracle,任何後續加工路線都要能回退比對這條基準線。
2. **第二階段**:路線 B 整數倍放大(建議先 2x = 1280×960),作為面向現代顯示器的預設可選項——工作量小、風險低、不牽動素材檔案,可做成執行期設定(視窗選單的「顯示縮放」選項),與語言切換(繁中/英文)、版本切換(1.3/1.5)並列在同一組使用者可調設定內。
3. **第三階段(選擇性)**:路線 C AI 超解析度放大,限定套用在低動態的大尺寸靜態素材(主選單背景、種族頭像、過場畫面),且比照 `rulebook/83` 的「別漏看資產檔」精神,**先窮舉哪些 asset 屬於這個範圍再動工**,輸出需人工抽樣比對原圖(尤其 dithering 區域),並在資產目錄清楚標示「AI 重建版,非原版逐位元組解碼」以與保真素材區隔。動畫序列與小尺寸圖示因一致性與幻覺風險,暫不列入此階段範圍(不是「不值得」而是「方法尚未成熟到可安全套用」,待有可靠的時間一致性處理法再重新評估)。
4. **第四階段(範圍限定)**:路線 D 人工重繪僅用於本專案原創、原版沒有對應物的新介面元素,不用於任何既有原版素材的替代。

三、四階段目前列為 `WORKLIST.md` 的後續追蹤項,不在本輪排入具體工時——但依專案宗旨,「後續追蹤」不等於「判定不值得」，待路線 B 的執行期縮放框架完成後,應重新排定路線 C 的窮舉清單與試作範圍。
