# 按鈕中文化(kick-off)—— 避免重蹈覆轍

> CLAUDE.md 特別要求:「按鈕的中文化一定要參考先前的專案經驗,避免重蹈覆轍」。
> 前作原文:`~/master-of-orion/docs/CJK-DEV-NOTES.md` §3 烘字 SOP、§4 踩雷、速查表。

## 1. 核心事實:按鈕的英文常是「圖」,不是「字」

MOO 系列的按鈕、面板標籤,英文字**烘進了 .lbx 的點陣圖**(gfx),不是可替換的字串。
這是所有按鈕中文化坑的根源:你翻譯字串表沒有用,因為那段英文根本不在字串表裡,而是圖的像素。

在 **MOO1(1oom, patch 既有引擎)** 的世界,因為不能改版權 .lbx,只能用 **overlay 疊字**:
每幀取樣鈕面底色蓋掉英文、再畫中文(SOP 見前作 §3)。這很脆弱,踩了一連串雷:

- t0/t3 widget 會在 callback 後**重畫 gfx**,把疊好的中文蓋掉(症狀:中文閃一下消失)→ 得改 mousearea。
- 疊在一次性 page2 背景快取的中文變空白 → 疊字必須在**每幀** draw_cb。
- 字級全域寫死 14px,大按鈕字太小 → 要**依按鈕高度自選字級**。

## 2. 本專案(ebiten 從零重寫)怎麼做才對

**我們掌控整條繪製管線,不必也不該照搬 overlay 疊字 hack。** 正確做法:

1. **按鈕拆成「背景 gfx + 標籤文字」兩層,由我們自己組。**
   - 背景:從 .lbx 讀按鈕底圖(**用不含英文的底框那一格**,或用純色/九宮格 nine-patch 自繪),不要用「已烘英文」的那一格。
   - 標籤:一律走 `lang.Get(key)` 取當前語言字串,用 TTF 在按鈕上**置中**畫出。
2. **先盤點:哪些按鈕的 .lbx 有「乾淨底框格」可用,哪些只有「已烘英文」格。**
   - 有乾淨格 → 直接畫底框 + 文字。
   - 只有已烘英文格 → 兩條路:(a) 取樣鈕面色蓋掉英文區再畫中文(沿用 overlay 思路,但只做一次進我們的 texture);(b) 自繪 nine-patch 按鈕完全取代該圖。優先 (b),更乾淨、可控。
3. **狀態(normal/hover/pressed/disabled)由我們的 widget 狀態機決定畫哪張底 + 文字色**,不依賴原引擎 t3 的自動切換(那正是 MOO1 的雷 §3.3)。

## 3. 字級規則(承襲 §4.6 教訓)

- 依按鈕**容器高度**自選字級,不要全域寫死。
- 建議門檻(依實際底圖高度校正):高鈕(標題/確認)用大字、工具列小鈕用小字。
- 中文用實際字高**垂直置中**於按鈕,不是貼底線(ebiten 有完整字體 metrics,可精準置中)。

## 4. 尺度一致(承襲 §4.5 教訓)

按鈕群 / 清單列的中文放大列距時,「列距 / 選擇框高 / 可視區總高」三者要一起換算,
否則清單破格(前作艦船設計元件清單踩過)。中文按鈕比英文寬,**排版要預留寬度**或改為自動換行 / 縮字。

## 5. 驗證(每個含按鈕的畫面都要做)

- docker build → xvfb 截圖 → Read 圖逐屏檢查:中文有沒有畫出、有沒有被底圖蓋掉、置中對不對、有沒有溢出。
- 特別檢查 hover/pressed 狀態切換時中文是否還在(MOO1 最常見的「閃一下消失」)。
