# 可行性評估總論(kick-off round 1)

> 本文是第一輪可行性研究的總結。細節分見同目錄 01~06。撰於 2026-07-02。

## 1. 核心結論

**可行,但要先認清工作量的真實分布。** 這不是「把一個能玩的遊戲翻成另一個語言」,而是:

1. **一小部分**:把 openorion2 現成的「資產解碼 + 存檔資料模型 + 檢視器 UI」搬到 go/ebiten。
2. **絕大部分**:從原版手冊 + 逆向,**從零重建整個回合制遊戲引擎**(經濟、科技、戰鬥、外交、AI)——openorion2 這塊**根本不存在**。

把這兩件事混為一談會嚴重低估時程。PLAN.md 據此分成「移植檢視器」與「重建遊戲」兩大階段。

## 2. openorion2 給我們什麼、缺什麼

| 拿到手的(可複用) | 缺的(要自建) |
|---|---|
| LBX 資產解碼器(容器 + RLE + palette) | 全部 gameplay 規則(回合/經濟/科技/戰鬥/AI) |
| 完整存檔資料 schema(Colony/Planet/Star/Ship/Player/Leader…) | 音樂/音效(宣稱有、實際零實作) |
| 資料枚舉/常數字典(技術≈200、建築、種族特性) | 可 CJK 的文字系統(現有 bitmap font 單 byte,無法中文) |
| 唯讀衍生公式(艦艇戰力、研究成本、行星產出) | 「開新遊戲」與所有會改變狀態的操作(現為 STUB) |
| retained-mode UI framework 邏輯 + 各檢視畫面佈局 | ebiten 繪圖/事件後端(現為 SDL) |

## 4. 最大風險與對策

- **風險:低估 gameplay 重建工作量。** 對策:PLAN 明確切「檢視器移植」vs「遊戲重建」;先交付可跑的檢視器建立信心,再逐系統長回合制引擎。
- **風險:規則正確性(MOO2 規則繁複、社群考據深)。** 對策:以原版手冊 + 1.5 手冊為權威;逆向 openorion2 既有公式常數當交叉驗證;疑點用第一性原理查證,不臆測(對齊 CLAUDE.md)。
- **風險:音樂/音效無參考。** 對策:單獨逆向 .lbx 內 XMI/音效格式,或先靜音佔位,排後續(完整性優先,不砍)。
- **風險:`~/master-of-magic` go/ebiten 參考缺失。** 對策:先以 ebiten 官方 + openorion2 架構推進;取得後回填。

## 5. 建議的第一個可交付里程碑

**「能在 ebiten 開視窗、讀玩家的正版 .lbx、載入一個存檔並顯示星系圖」** —— 這打通「資料層 → ebiten 畫面」全鏈路,
且完全複用 openorion2 已完成的部分,風險最低、最能驗證技術路線。之後才逐步把 STUB 換成真正的 gameplay。
