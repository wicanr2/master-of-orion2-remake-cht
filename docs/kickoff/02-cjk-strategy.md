# 中文化策略(kick-off)

> 目標:openorion2 → go/ebiten 重寫版,支援英文/繁體中文切換、完整中文化(含按鈕)。
> 本文以第一性原理重新推導策略,並吸收前作 MOO1(1oom)繁中化的實戰經驗。
> 前作經驗原文:`~/master-of-orion/docs/CJK-DEV-NOTES.md`(建議先讀)。

## 1. 與前作 MOO1 的根本差異(先講清楚,否則會照抄錯方法)

| 面向 | MOO1(1oom) | 本專案(openorion2 → ebiten) |
|---|---|---|
| 我們的角色 | **patch 既有 C 引擎**,不能碰版權 .lbx | **從零用 ebiten 重寫 renderer**,掌控整條繪製管線 |
| 中文渲染手法 | overlay 疊字(取樣底色蓋英文 + 每幀畫中文) | 直接在自己的 draw 流程用 TTF/點陣渲染中文,無須「蓋字 hack」 |
| 解析度 | 320×200 邏輯層,2× 合成層畫 CJK | ebiten 可原生高解析;320×200 底圖上採樣 + 中文用向量字直接畫 |
| 字串來源限制 | 受既有 lbxfont 多路徑牽制(§4.3/4.4 踩雷) | 自己設計文字管線,一條路徑到底,不會漏解碼 |

**結論**:MOO1 的 overlay 疊字 SOP **不要照搬**——那是「不能改引擎」的妥協。本專案自己畫,直接用 Unicode 文字 + TTF。
但 MOO1 的**踩雷教訓**(尤其「按鈕英文烘在圖裡」「字級/尺度換算」「密集標籤別用低解析直覺判斷」)**完全適用**,見 §4。

## 2. 文字來源分類(決定各自的中文化手法)

從 openorion2 移植時,遊戲文字有三種來源,處理方式不同:

| 文字來源 | 說明 | 中文化手法 |
|---|---|---|
| **程式硬編碼字串** | 我們自己 Go 程式碼裡的 UI 文字 | 走同一顯示層覆蓋(英文原文即 key);⚠️ 別假設「文字全在 LBX」——openorion2 也有 hardcode 佔位訊息 |
| **LBX 內字串資源** | 科技名/描述/種族/外交/事件等,存在版權 .lbx | 執行期讀出後,顯示前經覆蓋層換中文;譯文存 repo 的 `*.tsv`(只放譯文,不放原文版權內容) |
| **烘進 gfx 的英文**(按鈕/面板標籤) | 英文是圖的一部分,不是字串 | 擦底疊字 or 整圖替換,見 `03-button-cht.md` |

> ⚠️ 合規底線(承襲 patch-only 精神):版權文字(LBX 原文)不入 repo,只入庫「我們的譯文對照表」。玩家自備正版 .lbx,執行期讀取。

**架構採 mom 已驗證的「顯示層覆蓋 + 英文原文即 key」**(詳見 `08-mom-ebiten-cht-playbook.md` §1):

- **[HARD] 只在顯示層翻,不動資料層** —— openorion2 大量把英文字串當邏輯 key(從 LBX 讀出後比對/索引),改資料層會壞邏輯。
- **英文原文即 TSV 的 key**(非另編 id):同一表可相容 1.3/1.5 不同版本字串,查無 key 自動 fallback 英文。
- **組合字串**用 `TranslateFormat`:先翻模板字面(`"Cost %v"→"花費 %v"`)再 `Sprintf` 填值;佔位符數/序中英必一致。
- **[HARD] 開工第一步**:窮舉**所有**文字源(LBX 各類 + hardcode),各寫一支 dumper,用引擎自己的 reader dump 精確 key(漏一源 = 整類畫面英文;對齊 rulebook 83)。

## 3. 語言切換架構(en / zh 主選單可選)

> ⚠️ **與 mom 的差異**:mom 是「出中文包 or 英文包」,**無 runtime 切換**。但 CLAUDE.md 明確要求**主選單可即時切中/英** → 我們要做 runtime 切換。顯示層覆蓋天然支援:切語言 = 切「是否套用覆蓋層 + 換字型」。

- 顯示前一律經覆蓋層:`display(text)` → zh 時查表換中文(查無 fallback 英文)、en 時原樣回傳。
- 所有文字繪製都走這條;繪字管線把 CJK 當一等公民(見下 §「渲染」)。
- 字型隨語言切換:中文載 CJK 字型,英文可用原版點陣風格字型(見 `04-font-choice.md`)。

## 3.5 渲染要點(從零寫,取 mom 原理)

- **[HARD] supersample,不「小字放大」**:CJK 以 logical 字高 ×4 高解析 rasterize 再縮回,維持 320×200 座標系;不需獨立高解析合成層(細節 `08` §2)。
- **glyph 快取** key=`(rune, 字高)`;CJK 尺寸依**呼叫端字高**決定(否則行距重疊破版),基線比例約 0.82。
- **所有繪字路徑**(一般/描邊陰影/**量寬 MeasureTextWidth**)都要支援 CJK,否則置中/換行算錯。
- **[HARD] CJK 無空白 → 自寫逐字斷行**(無可斷點時至少切一 rune),否則整段中文被丟。

## 4. 從 MOO1 帶過來、本專案仍要守的教訓

- **4.1 別用低解析直覺判斷「塞不下 CJK」**:先算清楚底圖尺度 ↔ 顯示尺度的換算。ebiten 高解析下空間更充裕,但仍要用實機截圖校對密集標籤區。
- **4.5 清單三尺度一致**:列距、選擇框高、可視區總高要換算成同一尺度;中文放大列距時,容納上限跟著變 → 提早改捲動清單。
- **4.6 字級依容器高度自選**:大按鈕用大字、矮列用小字;不要全域寫死一個字級。
- **按鈕是最大坑**:見 `03-button-cht.md`,獨立成篇(CLAUDE.md 特別要求)。
- **驗證紀律**:每個畫面改完跑 docker build + xvfb 截圖 + Read 圖逐屏校對(破版/溢出/缺字),不靠瞪程式碼(對應 `04.7`/§5)。

## 5. openorion2 盤點回填(已確認)

已由 `01-openorion2-assessment.md` §6 證實:

- **LBX 字串涵蓋廣**:`TextManager` 管理主文字/事件訊息/種族名/艦名/星名/技能/科技名+描述/外交/help 全文,幾乎無硬編碼 → 譯文表規模不小,需系統性建立。
- **已有 i18n 骨架**:openorion2 內建 5 語言(英德法西義),`selectLanguage()` 重載字串+字型 → 中文可沿用此結構,新增為第 6 語言,資料源改外部 UTF-8。
- **字型是唯一結構障礙**:原 bitmap font 以 byte 值當 glyph 索引(單 byte,最多 256 字),**結構上無法 CJK** → 印證本文 §1 結論:必須自建文字系統,直接用 TTF + Unicode,反而免去 moo1 的 overlay 妥協。
- **待清點**:哪些按鈕/面板標籤是烘進 gfx(非字串)→ 見 `03-button-cht.md`,需逐畫面盤點。
